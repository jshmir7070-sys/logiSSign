이전 이슈 추적 결과

RESOLVED RLS 오버퍼미시브 정책 — 데이터 유출 위험
003_security_hardening.sql 마이그레이션으로 seals, document_files, document_deliveries, document_sign_fields, document_sign_responses 테이블 모두 agency_id/driver_id 기반 세밀한 RLS 정책이 적용되었습니다. USING(true) 완전 제거 확인.
RESOLVED 스토리지 버킷 public 설정
003_security_hardening.sql에서 UPDATE storage.buckets SET public = false WHERE id IN ('contracts', 'documents', 'education') 확인. 마이그레이션 순서대로 적용 시 최종적으로 private 설정됨.
RESOLVED 결제 빌링 키 PG사 검증
포트원 V2 API 연동 완료. payment/route.ts에서 https://api.portone.io/billing-keys/{billingKey}로 빌링 키 유효성 검증 후 DB 저장. 카드 번호 마스킹 처리 확인.
RESOLVED 기사 가입 API 통합 (driver-signup)
기존 link-driver, verify-invite 분리 엔드포인트를 driver-signup으로 통합. 초대코드 검증 → Auth 생성 → Driver 연결을 원자적으로 처리하며, 실패 시 Auth 롤백 포함. Rate limiting 적용 완료.
RESOLVED 누락 컬럼 + 인덱스 추가
driver_contract_periods에 rate_config(JSONB), amendment_id(UUID), memo(TEXT) 컬럼 추가 완료. education_records, contract_amendments, security_logs, notices, settlements 등 8개 성능 인덱스 추가.
PARTIALLY RESOLVED api-schemas.ts validateInput() 함수
Zod 스키마 정의는 완전하나, validateInput() 함수의 return문이 여전히 71행에서 잘려 있습니다. 다만 실제 API 라우트에서는 동작하고 있어 빌드 타임에는 문제없는 것으로 보이나, 코드 정리가 필요합니다.
PARTIALLY RESOLVED CSRF 토큰 보호
보안 헤더(HSTS, X-Frame-Options, CSP 등)는 완벽히 설정되었으나, 명시적 CSRF 토큰 검증 메커니즘은 아직 미구현입니다. Supabase JWT 기반 인증이 일정 수준 방어하지만, 쿠키 기반 세션에서는 추가 보호가 권장됩니다.
STILL OPEN 환경변수 시크릿 노출
web/.env.local에 SUPABASE_SERVICE_ROLE_KEY, SOLAPI_API_KEY, SOLAPI_API_SECRET, CRON_SECRET가 여전히 포함되어 있습니다. Git 히스토리에 커밋된 상태이므로, 키 로테이션이 반드시 필요합니다.
즉시 조치: 모든 키 재발급 + git filter-repo로 히스토리 정리
STILL OPEN 모바일 본인인증 (PASS/카카오) 미구현
mobile/services/identity.service.ts가 여전히 개발모드 패스쓰루 상태입니다. verified: true를 항상 반환하며 실제 PASS/카카오 인증 연동이 없습니다. 웹에서는 포트원 V2 본인인증이 구현되었으나, 모바일 앱에서는 미적용.
즉시 조치: 웹과 동일하게 포트원 V2 본인인증 WebView 연동, 서버 측 검증 결과 확인
STILL OPEN RPC 함수 권한 검증 부재
approve_amendment_with_period() RPC 함수 내부에 여전히 호출자 권한 검증 로직이 없습니다. SECURITY DEFINER로 실행되므로, API 레이어의 검증에만 의존하는 상태입니다.
권장: RPC 함수 내부에 auth.jwt() 기반 역할/소속 검증 추가

신규 발견 보안 이슈

NEW MEDIUM 포트원 API 토큰 캐싱 없음
payment/route.ts에서 매 결제 요청마다 PORTONE_API_SECRET으로 토큰을 새로 발급합니다. 토큰 만료 시간 검증이 없고, 불필요한 API 호출이 증가합니다.
권장: 토큰 캐싱 + 만료 시간 추적 구현
NEW MEDIUM 에러 메시지에서 포트원 내부 정보 노출
결제 API에서 포트원 에러 메시지를 '빌링키 검증 실패: ' + err.message로 클라이언트에 그대로 반환합니다. 포트원 API의 내부 구조가 노출될 수 있습니다.
권장: 에러 메시지 정제, 실제 에러는 서버 로그에만 기록
NEW MEDIUM AI extract-document API 에러 핸들러 미완성
extract-document/route.ts의 catch (err) 블록이 70행에서 코드가 중단되어 있습니다. 에러 발생 시 HTTP 응답 없이 타임아웃됩니다. 입력 크기 제한(50KB)은 정상 적용됨.
즉시 조치: catch 블록에 return NextResponse.json({ error: ... }, { status: 500 }) 추가
NEW LOW Rate Limiting 여전히 메모리 기반
rate-limit.ts가 여전히 Map 기반으로, Vercel Serverless 다중 인스턴스에서는 각각 독립적으로 동작합니다. 공개 엔드포인트(driver-signup 등)에 대한 분산 공격에 취약합니다.
권장: Upstash Redis 기반으로 전환
NEW LOW CSP unsafe-eval 유지
next.config.mjs에서 Daum Postcode 서비스를 위해 unsafe-inline과 unsafe-eval이 여전히 허용됩니다. XSS 공격 시 악용 가능성이 있습니다.
권장: nonce 기반 CSP로 전환 검토, Daum Postcode iframe 샌드박싱

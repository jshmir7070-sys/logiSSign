긴급 해결 필요 사항 (CRITICAL / HIGH)

CRITICAL RLS 오버퍼미시브 정책 — 데이터 유출 위험
seals, document_files, document_deliveries, document_sign_fields, document_sign_responses 테이블의 RLS 정책이 USING (true)로 설정되어 있어, 모든 인증 사용자가 다른 운영사/기사의 도장 이미지, 문서 파일, 서명 응답에 접근 가능합니다.
해결: 각 테이블에 agency_id 또는 owner_id 기반 RLS 정책 적용
CRITICAL 환경변수 시크릿 노출
web/.env.local에 SUPABASE_SERVICE_ROLE_KEY, SOLAPI_API_KEY, SOLAPI_API_SECRET, CRON_SECRET이 포함되어 있습니다. Git 히스토리에 포함되었을 경우 즉시 키 로테이션이 필요합니다.
해결: 모든 노출된 API 키 즉시 재발급, .env.local git 히스토리에서 제거 (git-filter-repo)
CRITICAL 스토리지 버킷 public 설정 불일치
002_storage_buckets.sql에서 contracts, documents, education 버킷이 public = true로 설정되어 있어, 누구나 계약서/문서/교육 증명서 파일을 다운로드할 수 있습니다. schema.sql에서는 false로 정의되어 있어 불일치합니다.
해결: 002_storage_buckets.sql에서 해당 버킷을 public = false로 수정
CRITICAL api-schemas.ts 파일 손상
src/lib/api-schemas.ts 라인 71-72의 validateInput() 함수 return문이 불완전합니다. 모든 API 입력 검증이 실패할 수 있습니다.
해결: 해당 함수의 return 구문 복구
HIGH CSRF 토큰 부재
모든 POST/PATCH/DELETE API 라우트에서 CSRF 토큰 검증이 없습니다. 쿠키 기반 인증(Supabase 세션)을 사용하므로, 악의적 사이트에서 계약서 발송, 결제 정보 변경 등을 유발할 수 있습니다.
해결: CSRF 토큰 미들웨어 구현 또는 SameSite=Strict 쿠키 + Origin 헤더 검증 추가
HIGH 빌링 키 저장 시 PG사 검증 없음
/api/payment/route.ts에서 클라이언트가 보낸 billingKey를 검증 없이 DB에 저장합니다. 결제 게이트웨이(Toss Payments 등)에서 발급받은 유효한 키인지 확인하지 않습니다.
해결: PG사 API로 빌링 키 유효성 검증 후 저장
HIGH 모바일 본인인증(PASS/카카오) 미구현
identity.service.ts에서 requestIdentityVerification()이 개발 모드 패스쓰루로, 항상 verified: true를 반환합니다. 계약서 서명의 법적 효력이 없습니다.
해결: PASS/카카오 인증 실제 연동 구현, 서버 측 인증 결과 검증
HIGH SECURITY DEFINER 함수 권한 검증 부족
approve_amendment_with_period() RPC 함수가 SECURITY DEFINER로 실행되지만, 호출자의 역할/소속 운영사 검증이 함수 내부에 없습니다. 또한 driver_contract_periods 테이블에 rate_config, amendment_id, memo 컬럼이 누락되어 있습니다.
해결: RPC 함수 내 권한 확인 로직 추가 + 누락 컬럼 마이그레이션

미완성 기능 상세

1. 모바일 — 푸시 알림 (push.service.ts)

파일이 12줄 주석만 존재하며 구현이 전혀 없습니다. registerPushToken(), addNotificationListeners(), unregisterPushToken(), getNavigationFromNotification() 모두 미구현 상태입니다. expo-notifications 패키지는 설치되어 있으나 사용되지 않습니다.
필요 작업: Expo Push Notification 전체 플로우 구현 (토큰 등록 → 서버 저장 → 수신 처리 → 딥링크 라우팅)

2. 모바일 — 서명 패드 (SignaturePad.tsx)

SVG 경로를 단순 base64 인코딩하여 시뮬레이션으로만 동작합니다. 실제 PNG/JPG 이미지가 아니며 PDF에 렌더링되지 않습니다. 전자서명법 준수를 위한 법적 효력이 없습니다.
필요 작업: react-native-signature-canvas 또는 ViewShot 기반으로 실제 이미지 서명 구현

3. 모바일 — 본인인증 (identity.service.ts)

console.log('[Identity] 인증 요청 — 개발모드 자동 통과')와 함께 항상 verified: true를 반환합니다. PASS, 카카오 인증 모두 실제 연동이 없습니다. TODO 주석만 존재합니다.
필요 작업: PASS/카카오 본인인증 서버 세션 생성 → WebView 인증 → 결과 콜백 처리 구현

4. Web — 결제 시스템 (payment API)

빌링 키 저장과 구독 충전 로직은 구현되어 있으나, PG사(Toss Payments 등) 실제 연동이 없고 빌링 키 검증이 빠져 있습니다. 실결제 테스트가 불가능한 상태입니다.
필요 작업: PG사 SDK 연동 + 빌링 키 발급/검증 + 웹훅 수신 처리

5. 전체 — 테스트 코드

Web, Mobile 모두 단위 테스트, 통합 테스트, E2E 테스트가 전혀 없습니다. 서비스 레이어, API 라우트, 컴포넌트 등 핵심 로직의 검증이 불가능합니다.
필요 작업: Jest/Vitest 설정 + API 라우트 테스트 + 서비스 레이어 테스트 + Cypress/Playwright E2E

6. DB — 누락 컬럼 및 인덱스

driver_contract_periods 테이블에 rate_config(JSONB), amendment_id(UUID), memo(TEXT) 컬럼이 누락되어 RPC 함수 실행 시 에러가 발생합니다. 또한 education_records, contract_amendments, security_logs 등에 성능 인덱스가 부족합니다.
필요 작업: ALTER TABLE 마이그레이션 + 추가 인덱스 생성

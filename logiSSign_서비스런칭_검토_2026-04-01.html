프로젝트 구조 요약

logiSSign은 라스트마일 배송업체(운영사/대리점)의 기사 정산, 전자계약, 교육관리를 디지털화하는 B2B SaaS 플랫폼입니다. 제공사(Provider) → 운영사(Agency) → 기사(Driver) 3계층 멀티테넌시 구조로, Web 포털에서 운영사가 관리하고, 모바일 앱에서 기사가 계약 서명 및 정산 확인을 합니다.

보안 이슈 상세

서비스 런칭 전 반드시 해결해야 할 보안 사항들입니다. 심각도별로 분류했습니다.

심각도이슈설명해결 방법
긴급 Middleware getSession() 사용 middleware.ts 70번 줄에서 getSession()을 사용 중. Supabase 공식 권장은 getUser()로, getSession은 JWT를 서버에서 재검증하지 않아 만료된 토큰이 통과될 수 있음. getUser()로 교체
긴급 CRON_SECRET 미설정 .env.local에 CRON_SECRET이 비어있음. Vercel CRON 엔드포인트(/api/cron/*)가 프로덕션에서 인증 없이 호출 가능. 단, 코드에서 production일 때 500 반환하는 방어 로직은 존재. 강력한 랜덤 시크릿 생성 후 Vercel 환경변수에 등록
긴급 CSP(Content Security Policy) 미적용 next.config.mjs에서 CSP가 주석 처리됨. XSS 공격에 취약. Kakao Postcode 호환을 위해 비활성화했지만, 적절한 도메인 화이트리스트로 해결 가능. CSP 설정 + Kakao 도메인 허용
높음 Rate Limiter 인메모리 방식 lib/rate-limit.ts가 메모리 기반. Vercel Serverless는 인스턴스마다 별도 메모리를 사용하므로 Rate Limit이 사실상 무력화됨. Vercel KV 또는 Upstash Redis로 교체
높음 은행 계좌정보 평문 저장 drivers 테이블에 bank_name, bank_account, account_holder가 평문으로 저장됨. 개인정보보호법 위반 소지. 암호화 저장 또는 마스킹 처리
높음 CORS Origin 하드코딩 NEXT_PUBLIC_APP_URL이 localhost:3000으로 설정. 프로덕션 도메인으로 변경 필요. Vercel 환경변수에 프로덕션 URL 설정
중간 Storage Bucket public 설정 contracts, documents, education, seals 버킷이 모두 public. 서명된 계약서와 기사 서류가 URL만 알면 접근 가능. settlements 외 나머지도 private + signed URL 방식 검토
중간 에러 메시지 노출 일부 API에서 Supabase 내부 에러 메시지가 클라이언트에 그대로 전달됨. DB 구조 정보 유출 가능. 프로덕션용 에러 메시지 래핑

인증/권한 아키텍처 평가

  • ✓
    RBAC 3계층 구현

    provider_admin / agency_admin / driver 역할 분리. app_metadata 기반으로 클라이언트 조작 불가.

  • ✓
    API 라우트 인증

    authenticateRequest(), authenticateAdmin(), authenticateCron() 3단계 인증 헬퍼. agency_id 필수 검증.

  • ✓
    RLS(Row-Level Security)

    20개 이상 정책으로 다른 운영사 데이터 접근 차단. auth.jwt()의 agency_id로 격리.

  • ✓
    Timing-Safe CRON 인증

    timingSafeEqual로 타이밍 사이드채널 공격 방지. 프로덕션에서 시크릿 미설정 시 500 반환.

  • !
    Middleware에서 getSession() → getUser() 전환 필요

    현재 미들웨어가 JWT를 서버에서 재검증하지 않음. Supabase 공식 권장 패턴으로 변경해야 함.

  • !
    서명된 계약서 Storage가 public

    계약서 PDF URL이 외부에서 접근 가능. private 버킷 + signedUrl로 변경 권장.

서비스 런칭 전 필수 액션 플랜

우선순위별로 분류한 태스크입니다. Phase 1은 런칭 전 반드시, Phase 2는 베타 기간 내, Phase 3은 정식 오픈 전 완료해야 합니다.

Phase 1: 즉시 해결 (1~3일)

  • CRON_SECRET 생성 — openssl rand -hex 32로 생성 → Vercel 환경변수 등록
  • middleware getUser() 전환 — getSession()을 getUser()로 교체
  • NEXT_PUBLIC_APP_URL — 프로덕션 도메인으로 변경
  • CSP 헤더 활성화 — Kakao, Supabase 도메인 화이트리스트 추가
  • 시드 데이터 투입 — 테스트용 운영사/기사/원청사 생성
  • Solapi API 키 발급 — SMS 발송 기능 활성화

Phase 2: 베타 준비 (1~2주)

  • 결제 시스템 연동 — Toss Payments 또는 포트원 PG 연동
  • Rate Limiter 교체 — Upstash Redis 기반으로 전환
  • Storage 접근 제어 — contracts/documents 버킷 private 전환
  • 은행정보 암호화 — 계좌번호 AES 암호화 또는 마스킹
  • 에러 모니터링 — Sentry 설정 (Web + Mobile)
  • 본인인증 연동 — PASS 또는 Kakao 인증 실제 연동
  • 모바일 앱 스토어 등록 — Apple Developer + Google Play Console

Phase 3: 정식 오픈 전 (2~4주)

  • 테스트 코드 작성 — 정산 계산, 계약 서명 등 핵심 로직 단위 테스트
  • Server Components 전환 — 데이터 페칭 페이지를 SSR로 최적화
  • Supabase Pro 전환 — DB/Storage 용량 확장, 일일 백업
  • 푸시 알림 완성 — Expo Notifications 호환 해결
  • 이메일 템플릿 — 계약 발송, 정산 알림 등 커스텀 이메일
  • DB 타입 공유 — 모노레포 packages/types로 통합
  • 정산 DB 제약조건 — 체크 제약/트리거로 계산 무결성 보장

Phase 4: 런칭 후 개선

  • 성능 최적화 — Lighthouse 점수 개선, 번들 사이즈 축소
  • A/B 테스트 — 랜딩 페이지, 전환율 최적화
  • 관리자 대시보드 고도화 — 실시간 데이터, 알림 시스템
  • API 문서화 — Swagger/OpenAPI 자동 생성
  • 다국어 지원 — 추후 해외 시장 진출 시

최종 종합 평가

logiSSign은 핵심 비즈니스 로직(정산, 전자계약, 교육)이 잘 구현된 MVP입니다. 26개 서비스 모듈, 33개 DB 테이블, Web + Mobile 양방향 포탈 구조는 상당한 완성도를 보입니다. 특히 한국 시장 맞춤형 기능(원천징수 3.3%, 한글 도장, 한국 주소 검색, SMS)이 잘 갖춰져 있습니다.

다만 프로덕션 준비도 55%로 평가한 것은 결제 시스템 부재, 보안 설정 미비(CSP, CRON_SECRET, 스토리지), 테스트 코드 없음 등이 서비스 안정성에 직접적인 위험 요소이기 때문입니다. Phase 1~2를 집중적으로 수행하면 2~3주 내 베타 런칭이 가능한 수준입니다.

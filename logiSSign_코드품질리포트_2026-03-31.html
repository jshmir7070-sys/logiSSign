🔴 Critical Issues 6건

CRITICAL verify/route.ts 파일 잘림 (Truncated)
web/src/app/(public)/verify/route.ts
계약서 진위확인 API 라우트의 소스코드가 중간에 잘려 있어 서버 빌드 시 파싱 에러가 발생합니다. 핵심 공개 기능(문서번호/인증코드 기반 진위확인)이 완전히 동작하지 않는 상태입니다.
수정: verify/route.ts를 완전한 구현으로 재작성 필요. 검증 로직, 응답 포맷, 에러 핸들링 포함.
CRITICAL 모든 /api/* 라우트가 미들웨어 인증을 우회
web/src/middleware.ts
미들웨어의 matcher 설정이 /(dashboard) 그룹만 보호하고, /api/* 경로를 포함하지 않습니다. 결과적으로 /api/contracts, /api/settlements 등 민감한 API가 미들웨어 레벨의 세션 검증 없이 노출됩니다. 각 라우트 핸들러에서 개별적으로 supabase.auth.getUser()를 호출하지만, 일부 라우트(특히 CRON)는 이 검증이 누락되어 있습니다.
수정: middleware.ts matcher에 /api/((?!cron|public|webhook).*) 패턴 추가. CRON은 별도 시크릿 인증, 나머지 API는 미들웨어에서 세션 검증.
CRITICAL CRON_SECRET 미설정 시 인증 우회
web/src/lib/api-auth.ts
authenticateCron()에서 CRON_SECRET 환경변수가 미설정이면 빈 문자열로 fallback되어, 공격자가 Authorization: Bearer (빈 값)으로 요청하면 인증을 통과할 수 있습니다. 또한 timing-safe 비교 전에 길이 체크(a.length !== b.length)가 있어 시크릿의 길이가 타이밍 사이드채널로 노출됩니다.
수정: CRON_SECRET 미설정 시 항상 403 반환. 길이 비교 제거 후 동일 길이 패딩 적용, 또는 Node.js crypto.timingSafeEqual만 사용.
CRITICAL 4개 테이블 schema.sql 누락
supabase/schema.sql vs 서비스 코드
서비스 코드에서 사용하는 4개 테이블이 schema.sql에 정의되어 있지 않습니다: contract_amendments, driver_contract_periods, contract_verification_logs, integrity_check_results. 마이그레이션으로 추가되었을 수 있으나, 기준 스키마 파일과 동기화되지 않아 새 환경 구축이나 스키마 감사에 문제가 됩니다.
수정: schema.sql에 4개 테이블 DDL 추가, 또는 supabase db dump로 최신 스키마 동기화.
CRITICAL contracts 테이블 컬럼 3개 누락
supabase/schema.sql — contracts
서비스 코드가 참조하는 signed_pdf_hash, document_number, verification_code 컬럼이 schema.sql의 contracts 테이블에 없습니다. 무결성 검사, 진위확인, 감사추적인증서 등 핵심 기능이 이 컬럼에 의존합니다.
수정: schema.sql에 해당 컬럼 추가. 마이그레이션 이력 확인 후 동기화.
CRITICAL identity.service.ts 스텁 구현
web/src/services/identity.service.ts
본인인증 서비스가 실제 PASS/카카오 API를 호출하지 않고 가짜 성공 응답과 더미 URL을 반환합니다. 프로덕션에서 본인인증 없이 전자서명이 완료되는 심각한 보안 이슈입니다.
수정: 실제 본인인증 제공사(PASS, 카카오) API 연동 구현 또는, 최소한 스텁임을 런타임에서 감지해 프로덕션 배포를 차단.

🟠 High Issues 8건

HIGH N+1 쿼리 — 무결성 검사
web/src/services/integrity-check.service.ts:82-166
runIntegrityCheck()에서 서명 완료 계약서를 전체 조회한 뒤, for 루프 안에서 계약 건마다 contract_signatures 개별 조회 + Storage PDF 다운로드를 수행합니다. 계약 1,000건이면 최소 2,000회 추가 요청이 발생합니다.
수정: contract_signatures를 계약 ID 배열로 일괄 조회(.in('contract_id', ids))한 뒤 Map으로 변환. PDF 다운로드는 Promise.allSettled로 병렬화 (동시성 제한 적용).
HIGH 에러 메시지 외부 노출
web/src/app/api/*/route.ts (다수)
API 라우트의 catch 블록에서 error.message를 그대로 JSON 응답에 포함합니다. 내부 구현 세부사항(테이블명, 컬럼명, Supabase 에러 코드)이 클라이언트에 노출되어 공격 표면이 넓어집니다.
수정: 프로덕션에서는 generic 에러 메시지 반환, 상세 에러는 서버 로그에만 기록. process.env.NODE_ENV === 'development'일 때만 상세 메시지 포함.
HIGH 사용자 메타데이터 미검증
web/src/app/(auth)/signup/page.tsx
회원가입 시 user_metadata에 입력된 agency_name, role 등을 서버사이드 검증 없이 Supabase Auth에 전달합니다. 클라이언트에서 role을 'admin'으로 조작하면 권한 상승이 가능합니다.
수정: 회원가입 API 라우트를 별도로 만들어 서버에서 역할 검증. user_metadata.role은 서버에서만 설정 가능하도록 변경.
HIGH 4개 테이블 RLS 미적용
supabase/schema.sql
contract_amendments, driver_contract_periods, contract_verification_logs, integrity_check_results 테이블에 RLS(Row Level Security)가 활성화되지 않았습니다. Admin 클라이언트 외에 anon/authenticated 키로 전체 데이터 접근이 가능합니다.
수정: 각 테이블에 ALTER TABLE ... ENABLE ROW LEVEL SECURITY 적용 및 agency_id 기반 RLS 정책 추가.
HIGH contract_signatures 컬럼 누락
supabase/schema.sql — contract_signatures
서비스 코드가 참조하는 audit_log, audit_certificate_url 컬럼이 schema.sql에 정의되지 않았습니다. 감사추적인증서 생성과 무결성 검사의 해시 체인 검증이 이 컬럼에 의존합니다.
수정: schema.sql 동기화 또는 ALTER TABLE 마이그레이션 확인.
HIGH timestamp_hash 검증 로직 버그
web/src/services/verification.service.ts (약 line 344)
verifyContractIntegrity()에서 timestamp_hash를 재계산한 뒤 !!expectedTsHash로만 확인합니다. 이는 해시가 존재하는지만 체크하고, 실제 DB에 저장된 값과 비교하지 않습니다. 타임스탬프 구성 요소가 변조되어도 탐지 불가능합니다.
수정: 타임스탬프 해시를 DB에 별도 컬럼으로 저장하고, 재계산 값과 비교 검증 로직 추가.
HIGH SQL 인젝션 위험 — .or() 필터
web/src/services/contract.service.ts
계약서 목록 조회에서 검색어를 .or() 필터에 직접 문자열 연결로 전달합니다. Supabase JS 클라이언트가 내부적으로 이스케이프하지만, PostgREST 필터 구문에서 특수문자((, ), ,)가 논리 구조를 변경할 수 있습니다.
수정: 검색어를 사전에 sanitize하거나, .ilike() 개별 필터로 분리하여 사용.
HIGH 정산 VAT 반올림 오류
web/src/services/settlement.service.ts
정산 계산에서 부가세(10%)와 원천징수세(3.3%)를 부동소수점 연산으로 처리합니다. 금액이 클 경우 1원 단위 오차가 누적되어 세금계산서와 불일치가 발생할 수 있습니다. 원천징수세율 3.3%가 하드코딩되어 세법 변경 시 수정이 필요합니다.
수정: Math.floor() 또는 Math.round()로 원 단위 절사. 세율을 환경변수 또는 설정 테이블로 분리.

🟡 Medium Issues 10건

MEDIUM Zod/런타임 입력 검증 부재
web/src/app/api/*/route.ts (전체)
API 라우트에서 request.json()으로 받은 입력을 타입 단언(as)으로만 처리하고, 런타임 스키마 검증(Zod, Yup 등)을 수행하지 않습니다. 잘못된 타입이나 누락 필드가 서비스 레이어까지 전파되어 예측 불가능한 오류를 발생시킵니다.
수정: 각 API 라우트에 Zod 스키마 도입. z.object({ contractId: z.string().uuid() }) 패턴으로 입력 검증.
MEDIUM Rate Limiting 미적용
web/src/middleware.ts, web/src/app/api/*
API 라우트에 요청 빈도 제한이 없습니다. 특히 무결성 검사(/api/integrity-check), 진위확인(/api/verify), 본인인증(/api/identity) 등 비용이 높은 엔드포인트가 DoS 공격에 취약합니다.
수정: Vercel Edge Middleware 또는 upstash/ratelimit 라이브러리로 IP/사용자 기반 rate limiting 적용.
MEDIUM 7개 인덱스 누락
supabase/schema.sql
자주 조회되는 패턴에 대한 인덱스가 부족합니다: contracts(agency_id, status), contracts(document_number), contracts(verification_code), contract_signatures(contract_id), settlements(agency_id, period_start), subscriptions(agency_id, status), payment_history(subscription_id).
수정: 위 7개 복합/단일 인덱스를 마이그레이션으로 생성.
MEDIUM 무결성 검사 admin 권한 미체크
web/src/app/api/integrity-check/route.ts:25
수동 무결성 검사 API에 TODO: 관리자 권한 확인 주석만 있고 실제 role 검증이 없습니다. 로그인한 모든 사용자(일반 대리점, 기사)가 전체 계약서 무결성 검사를 실행하거나 다른 대리점의 검사 이력을 조회할 수 있습니다.
수정: user_metadata.role === 'admin' || 'provider' 체크 추가, 또는 별도 admin 역할 테이블 기반 권한 검증.
MEDIUM signed-pdf.service.ts 이중 PDF 처리
web/src/services/signed-pdf.service.ts
PDF 생성 과정에서 새 PDF를 만들면서 동시에 기존 PDF를 로드하여 페이지를 복사하는 이중 처리가 발생합니다. 메모리 사용량이 2배가 되며, 대용량 계약서에서 OOM(Out of Memory)이 발생할 수 있습니다.
수정: 기존 PDF에 직접 검증 페이지를 추가하는 방식으로 리팩토링.
MEDIUM 감사추적 silent 에러 무시
web/src/services/integrity-check.service.ts:151, 243
감사추적 해시 체인 검증에서 catch { /* ignore */ }로 에러를 무시합니다. audit_log 컬럼이 존재하지 않는 경우를 대비한 것이지만, 실제 데이터 무결성 위반도 함께 무시될 수 있습니다.
수정: 에러를 구분하여 "컬럼 미존재"만 무시, 그 외 에러는 reasons에 추가하거나 로깅.
MEDIUM 포털 페이지 다수 파싱 에러
web/src/app/(dashboard)/portal/* (amendments, contracts/list, contracts/send, verify)
기사 포털의 여러 페이지가 불완전한 상태입니다 — JSX 태그가 닫히지 않거나 코드가 중간에 끊어져 있습니다. 빌드 시 해당 페이지들이 실패하여 기사 포털 전체 기능이 동작하지 않습니다.
수정: 각 포털 페이지의 JSX를 완성. 빌드 테스트로 파싱 에러 확인.
MEDIUM Base64 서명 이미지 미검증
web/src/services/signed-pdf.service.ts
서명 이미지를 Base64 문자열로 받아 PDF에 삽입할 때, 유효한 이미지 데이터인지 검증하지 않습니다. 악의적인 페이로드가 포함된 Base64 문자열이 메모리를 과도하게 사용하거나 pdf-lib 파서에 예기치 않은 동작을 유발할 수 있습니다.

수정: 계약의 agency_id와 기사의 소속 agency_id를 교차 검증하는 로직 추가.
MEDIUM 회원가입 폼 중복 검증

수정: 회원가입 전 서버사이드에서 사업자번호/이메일 유니크 체크. DB unique constraint도 추가.

Phase 1 — Critical (즉시 수정)

Week 1
Critical C-1. RLS 정책이 user_metadata 기반 — 사용자가 변조 가능
supabase/schema.sql — 다수 RLS 정책
RLS 정책이 auth.jwt()->'user_metadata'->>'agency_id'를 사용합니다. Supabase의 user_metadata는 클라이언트에서 supabase.auth.updateUser()로 수정 가능합니다. 악의적 사용자가 agency_id를 다른 대리점 ID로 변경하면 타 대리점 데이터에 접근할 수 있습니다.
-- 현재 (위험) CREATE POLICY "agencies_own" ON agencies FOR ALL USING (id = (auth.jwt()->'user_metadata'->>'agency_id')::uuid); -- 수정 방안: app_metadata 사용 (서버에서만 설정 가능) CREATE POLICY "agencies_own" ON agencies FOR ALL USING (id = (auth.jwt()->'app_metadata'->>'agency_id')::uuid);
수정: 모든 RLS 정책을 app_metadata 기반으로 변경. 회원가입 시 서버 Admin API로 app_metadata 설정. user_metadata는 표시용으로만 사용.
Critical C-2. auth.ts ↔ API 간 메타데이터 불일치 (app_metadata vs user_metadata)
lib/auth.ts:22 vs app/api/*/route.ts
lib/auth.ts는 app_metadata에서 role을 추출하고, API 라우트는 user_metadata에서 추출합니다. 두 소스가 다를 경우 권한 검사 결과가 일관되지 않아, 특정 경로에서 인가가 우회될 수 있습니다.
// lib/auth.ts (클라이언트) const role = user.app_metadata?.role // ← app_metadata // api-auth.ts (서버) const role = user.user_metadata?.role // ← user_metadata
수정: 전체 코드베이스에서 app_metadata로 통일. 회원가입 시 supabase.auth.admin.updateUserById()로 app_metadata에 role, agency_id 설정.
Critical C-3. 본인인증 서비스 개발모드 자동 통과
services/identity.service.ts:82-88, 120-135
API 키가 설정되지 않으면 본인인증이 자동으로 성공 처리됩니다. "dev_"로 시작하는 세션은 무조건 인증 통과. 프로덕션에서 환경변수 누락 시 누구나 타인 명의로 계약 서명이 가능합니다.
수정: 프로덕션에서 API 키 미설정 시 500 에러 반환. 개발모드에서도 명시적 플래그(ALLOW_DEV_IDENTITY=true) 없이는 통과 불가.
Critical C-4. contracts_driver_update RLS 정책 자기 참조 — 무력화
supabase/schema.sql:464-473
계약서 수정 방지 정책이 content = (SELECT content FROM contracts c2 WHERE c2.id = contracts.id)로 구현되어 있습니다. UPDATE 시 자기 자신과 비교하므로 항상 true — 기사가 계약서 내용을 변경할 수 있습니다.
-- 현재 (무력화됨) WITH CHECK ( content = (SELECT content FROM contracts c2 WHERE c2.id = contracts.id) ) -- 수정: 기사는 signature 관련 컬럼만 UPDATE 허용 CREATE POLICY "contracts_driver_sign" ON contracts FOR UPDATE USING (driver_id = auth.uid()) WITH CHECK ( -- content, template_id 등 핵심 필드 변경 불가 content IS NOT DISTINCT FROM (SELECT c.content FROM contracts c WHERE c.id = contracts.id) AND template_id IS NOT DISTINCT FROM (SELECT c.template_id FROM contracts c WHERE c.id = contracts.id) );
수정: UPDATE 트리거로 immutable 필드(content, template_id, document_number) 변경 시 에러 발생하도록 강제. RLS는 서명 관련 필드만 허용.

Phase 2 — High (1주 내 수정)

Week 1-2
High H-1. 비밀번호 복잡성 미검증 — 8자 이상만 확인
api/auth/signup/route.ts:16, portal/signup/page.tsx
"12345678" 같은 단순 비밀번호 허용. 대리점 관리자 계정이 탈취당하면 전체 기사 데이터와 정산 정보가 노출됩니다.
수정: Zod 스키마에 .regex(/^(?=.*[a-z])(?=.*[A-Z])(?=.*\d)(?=.*[@$!%*?&]).{8,}$/) 추가. 클라이언트에도 동일 규칙 적용.
High H-2. In-Memory Rate Limiter — 서버리스 환경에서 무력화
lib/rate-limit.ts:8-26
Vercel 서버리스에서 인스턴스별 메모리 분리. 공격자가 여러 인스턴스에 분산 요청하면 rate limit 우회. 공개 엔드포인트(/api/verify)가 특히 취약.
수정: @upstash/ratelimit + Upstash Redis로 교체. 글로벌 분산 rate limiting 보장.
High H-3. X-Forwarded-For 헤더 스푸핑으로 IP 기반 제한 우회
모든 API 라우트 (verify, signup, amendments 등)
클라이언트가 X-Forwarded-For 헤더를 임의로 설정하면 다른 IP로 인식되어 rate limiting, 감사 로그가 무력화됩니다.
수정: Vercel의 request.headers.get('x-vercel-forwarded-for') 또는 request.ip 사용. 신뢰할 수 있는 프록시 헤더만 참조.
High H-4. Service Role로 모든 API 쿼리 — RLS 전면 우회
api/amendments/route.ts, api/contracts/*/route.ts 등
API 라우트에서 createAdminSupabaseClient()(service_role)를 사용해 쿼리 실행. RLS 정책이 작성되어 있지만 service_role은 모든 정책을 우회합니다. 수동 agency_id 체크에 의존하므로, 체크 누락 시 데이터 유출.
수정: 일반 데이터 조회는 createServerSupabaseClient()(anon key + 사용자 JWT)로 RLS 적용. service_role은 시스템 작업(CRON, 관리자)에만 제한.
High H-5. contract_templates 테이블 CUD 정책 누락
supabase/schema.sql:564-570
SELECT 정책만 존재. INSERT/UPDATE/DELETE 정책이 없어 RLS 기본 동작(거부)에 의존하지만, service_role 사용 시 무방비.
수정: provider_admin만 INSERT/UPDATE 가능한 명시적 정책 추가. DELETE는 완전 차단(soft delete 패턴).
High H-6. SMS 메시지에 사용자 입력 직접 삽입 — SMS 인젝션
services/sms.service.ts:119-129
계약서 제목(contractTitle)이 검증/이스케이프 없이 SMS 본문에 삽입됩니다. 악의적 제목에 피싱 URL이나 특수문자 포함 가능.
수정: 계약서 제목 길이 제한(50자) + 허용 문자 필터링(한글/영문/숫자/공백만). SMS 본문 최대 길이 검증.
High H-7. .env.local 커밋 방지 미확인
.env.local, .gitignore
Service Role Key가 .env.local에 있으며, .gitignore에 포함되어야 합니다. 실수로 커밋되면 DB 전체 접근 가능.
수정: .gitignore 확인 + git-secrets 또는 pre-commit hook으로 시크릿 커밋 차단. 기존 키 로테이션.

Phase 3 — Medium (2주 내 수정)

Week 2-3
Medium M-1. 진위확인 코드 브루트포스 보호 부족
api/verify/route.ts, services/verification.service.ts
8자리 코드(약 2.8조 조합)에 IP당 10회/분 제한. 분산 공격 시 Redis 미적용으로 무력화.
수정: Redis rate limit 적용 후 3회 연속 실패 시 CAPTCHA 요구. 5회 실패 시 15분 차단.
Medium M-2. CRON 인증 개발모드 우회
lib/api-auth.ts:100-113
NODE_ENV가 'production'이 아니면 CRON_SECRET 없이 무결성 검사, 구독 갱신 등 CRON 작업 실행 가능.
수정: 모든 환경에서 CRON_SECRET 필수화. 로컬 개발은 .env에 테스트용 시크릿 설정.
Medium M-3. timingSafeEqual 비교 전 길이 체크 정보 누출
lib/api-auth.ts:119-130
authHeader.length === expected.length를 별도로 체크하여 시크릿 길이 정보가 타이밍 차이로 노출될 수 있습니다.
수정: 256바이트 패딩 후 timingSafeEqual만 사용. 별도 길이 비교 제거.
Medium M-4. 인증코드 유니크 검증 실패 시 중복 코드 사용
services/verification.service.ts:230-241
5회 재시도 후에도 유니크하지 않으면 그대로 사용. 대량 계약 발생 시 코드 충돌 가능.
수정: DB unique constraint + 재시도 실패 시 에러 반환. 코드 길이를 10자리로 확장 고려.
Medium M-5. 보안 이벤트 감사 로깅 부재
전체 API 라우트
인증 실패, 권한 거부, 비정상 접근 등 보안 이벤트에 대한 구조화된 로깅이 없습니다. 사고 발생 시 추적 불가.
수정: lib/security-logger.ts 모듈 생성. 인증 실패, CRON 접근, 데이터 변경 등 이벤트를 Supabase security_logs 테이블에 기록.
Medium M-6. 가격/플랜 조작 방어 미비
portal/signup/page.tsx → api 연동
결제 금액이 클라이언트에서 계산되어 서버로 전송됩니다. 중간에서 monthly_fee를 0으로 변조하면 유료 플랜을 무료로 사용할 수 있습니다.
수정: 서버에서 plan + billing_cycle 기반으로 금액 재계산. 클라이언트 전송 금액 무시.

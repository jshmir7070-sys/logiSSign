1. API 라우트 인증 완전 누락

모든 API 라우트 (/api/amendments, /api/contracts/send, /api/contracts/list)에 인증(auth) 검증이 전혀 없습니다. 요청 body의 agencyId를 그대로 신뢰하므로, 누구든 타 대리점의 계약서를 발송하거나 변경할 수 있습니다.

// 현재 — 인증 없음
export async function POST(req: NextRequest) {
  const { agencyId, driverIds, ... } = await req.json();
  // agencyId를 그대로 사용 → 위험

// 수정 방안
const supabase = createServerSupabaseClient();
const { data: { user } } = await supabase.auth.getUser();
const agencyId = user?.user_metadata?.agency_id;
if (!agencyId) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });

2. 계약 변경 승인 시 트랜잭션 미사용 (레이스 컨디션)

Amendment 승인(PATCH) 시 "기존 기간 만료 → 신규 기간 생성"이 2개의 별도 쿼리로 실행됩니다. 2번째 쿼리 실패 시 기사에게 활성 계약 기간이 사라집니다.

// 현재: 별개 쿼리 2개 (원자성 없음)

// 수정: Supabase RPC 또는 PostgreSQL Function으로 트랜잭션 래핑
await supabase.rpc('approve_amendment_with_period', { ... });

3. RLS 미적용 핵심 테이블 6개

다음 테이블에 RLS가 꺼져 있어, 인증된 사용자라면 누구든 전체 데이터에 접근 가능합니다.

agencies 대리점 전체 정보
principals 카테고리/단가 설정
contract_templates 계약서 템플릿
settlement_rules 정산 규칙
deduction_items 공제 항목
incentive_rules 인센티브 규칙

특히 agencies와 principals는 사업자번호, 계좌정보, 단가 설정 등 민감 정보를 포함하므로 반드시 RLS 적용이 필요합니다.

💬 총평

전체적으로 아키텍처와 비즈니스 로직 설계가 잘 되어 있습니다. 서비스 레이어 분리, FieldConfig 기반 동적 설정, 계약 변경 워크플로우, CRON 기반 갱신 관리 등 택배 대리점 정산 플랫폼에 필요한 핵심 기능이 빠짐없이 구현되어 있습니다.

가장 시급한 문제는 API 인증과 RLS입니다. 현재 상태로 배포 시 타 대리점 데이터 접근이 가능하므로, 운영 전에 반드시 해결해야 합니다. 두 번째는 트랜잭션 없는 다단계 DB 업데이트로, 데이터 무결성 문제가 발생할 수 있습니다.

이 두 가지를 해결하면 베타 출시에 충분한 수준이며, 나머지 Medium/Low 이슈는 점진적으로 개선해 나가면 됩니다.

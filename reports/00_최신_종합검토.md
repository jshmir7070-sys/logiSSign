## 1. 요약

- 현재 상태는 `내부 데모 가능 / 파일럿 직전 안정화 단계`다.
- 웹 포털, 관리자, 모바일 기사앱의 주요 화면과 업무 흐름은 대부분 구현되어 있다.
- 이번 정리로 AI 프롬프트는 `web/src/lib/ai-prompts.ts` 한 곳으로 통합했고, 오래된 검토 산출물은 `reports/` 기준 문서 한 장으로 정리했다.
- 파일럿 전 핵심 보완점은 `가입/결제 일관성`, `signed URL 저장 전략`, `무결성 검증 정합성`, `기사 가입 원자성`, `분산 rate limiting` 다섯 가지다.

## 3. 동작 중인 기능

- 포털: 대시보드, 기사 관리, 계약 발송, 정산 규칙/생성/이력, 세금계산서, 문서함, 공지, 설정
- 관리자: 대리점 관리, 매출/과금, 템플릿 관리, 감사로그, 서버상태, 공지
- 모바일: 기사 회원가입, 로그인, 계약 확인/서명, 정산 확인, 공지, 교육, 프로필
- 외부 연동: PortOne, Solapi, OpenAI, Sentry, Expo Push 코드 경로가 모두 존재
- 보안 기반: Supabase Auth, RLS, MFA, CSRF, 서버 측 role 검증 구조가 갖춰져 있다

## 4. 부분 완료 또는 미완료 기능

- 가입 결제 플로우는 아직 완전히 서버 중심으로 정리되지 않았다.
- 카드 결제는 백엔드 API가 있지만 `web/src/app/portal/signup/page.tsx` 안에 TODO와 브라우저 직접 `subscriptions` insert가 남아 있다.
- signed PDF와 감사증명서는 생성되지만, 만료형 signed URL을 영구 링크처럼 저장하고 있다.
- 무결성 검사는 signed URL 형식을 아직 완전히 따라가지 못한다.
- 운영 보안 측면에서 rate limiting이 메모리 기반이라 다중 인스턴스 환경에선 한계가 있다.

## 5. 실제 버그 및 운영 리스크

### 5-1. 기사 가입 기존 row 연결 실패를 조용히 놓칠 수 있음

- 파일: `web/src/app/api/auth/driver-signup/route.ts`
- 현상: 미리 만들어 둔 `drivers` row가 있을 때 update 에러를 명확히 반환하지 않으면, Auth 계정은 생겼지만 driver row 연결은 실패한 반쪽 상태가 남을 수 있다.
- 영향: 실 대리점 파일럿에서 기사 가입 실패가 재현돼도 원인 추적이 어렵다.

### 5-2. signed PDF URL을 영구 링크처럼 저장하고 있음

- 파일: `web/src/services/signed-pdf.service.ts`
- 현상: `createSignedUrl(..., 1년)` 결과를 `contracts.signed_pdf_url`에 그대로 저장한다.
- 영향: 링크 만료 후 포털과 모바일은 죽은 URL을 계속 신뢰하게 되고, 재생성/재발급 흐름이 꼬일 수 있다.

### 5-3. 무결성 검사가 예전 public URL 패턴을 기준으로 동작함

- 파일: `web/src/services/integrity-check.service.ts`
- 현상: `extractStoragePath()`가 `/storage/v1/object/public/contracts/...` 패턴만 처리한다.
- 영향: signed URL 기반 저장으로 바뀐 뒤 PDF hash 검증이 조용히 빠질 수 있다.

### 5-4. 가입 UI가 결제 백엔드를 우회하는 경로가 남아 있음

- 파일: `web/src/app/portal/signup/page.tsx`
- 현상: 가입 완료 직후 브라우저에서 `subscriptions`를 직접 insert하고, 카드 결제 분기에는 TODO가 남아 있다.
- 영향: 가입 성공, 결제 실패, 구독 데이터 불일치 같은 운영 이슈가 생길 수 있다.

### 5-5. 문자열/주석 인코딩 드리프트가 일부 소스에 남아 있음

- 파일 예시: `web/src/services/signed-pdf.service.ts`, `web/src/services/integrity-check.service.ts`, 과거 테스트/문서 일부
- 현상: 일부 한국어 문자열과 주석이 깨진 상태로 남아 있어 유지보수성이 떨어진다.
- 영향: 운영 중 치명 버그는 아니지만, QA와 후속 개발 속도를 늦춘다.

## 6. 보안 위험

### 6-1. 메모리 기반 rate limiter

- 파일: `web/src/lib/rate-limit.ts`
- 현상: 현재는 인메모리 `Map` 기반이다.
- 영향: 다중 인스턴스/서버리스 환경에서는 글로벌 throttling이 되지 않는다.
- 권장: Upstash Redis 같은 분산 저장소 기반으로 교체

### 6-2. 가입/결제 흐름의 서버 권한 경계가 완전히 닫히지 않음

- 파일: `web/src/app/portal/signup/page.tsx`, `web/src/app/api/payment/route.ts`
- 현상: 브라우저가 구독 row를 직접 만들고, 서버 결제 성공/실패 상태와 하나의 트랜잭션으로 묶여 있지 않다.
- 영향: 과금/권한/플랜 상태 불일치가 생길 수 있다.

### 6-3. 만료형 signed URL을 데이터 원본처럼 저장

- 파일: `web/src/services/signed-pdf.service.ts`, `web/src/services/audit-certificate.service.ts`
- 현상: 파일 경로가 아니라 URL 토큰 자체를 DB에 저장한다.
- 영향: 토큰 만료와 URL 유출을 동시에 관리하기 어렵다.

## 7. 코드 품질 이슈

- `web/src/app/portal/signup/page.tsx`는 한 파일에 가입, 본인인증, 플랜 선택, 결제 UI, 직접 DB insert 로직까지 몰려 있어 분리 필요성이 크다.
- `web/src/services/signed-pdf.service.ts`는 일반 계약, PDF형 계약, 관공서 양식 계약의 업로드/URL 저장 로직이 세 번 반복된다.
- AI 프롬프트는 이번 작업으로 공용화했지만, 나머지 한글 주석/문자열 인코딩은 후속 정리가 필요하다.
- 관리자 템플릿 화면은 `alert()` 중심 피드백이 많아 운영 UX가 투박하다.

## 8. 테스트 공백

- AI 라우트는 기존에 회귀 테스트가 거의 없었고, 이번 작업에서 프롬프트/파서 단위 테스트만 추가했다.
- 가입 → 결제 → 구독 반영 → 계약 발송 → 모바일 서명 → signed PDF → 무결성 검사 전체 E2E는 아직 자동화가 없다.
- signed URL 만료, audit certificate URL 만료, integrity-check storage path mismatch 같은 운영성 케이스도 테스트가 없다.

## 9. 파일럿 전 필수 수정 우선순위

1. 가입/결제 플로우를 서버 중심으로 정리하고 `subscriptions` 직접 insert를 제거
2. `contracts.signed_pdf_url`, `audit_certificate_url`에는 URL이 아니라 storage path를 저장
3. `integrity-check.service.ts`를 signed URL 저장 전략과 일치하도록 수정
4. `driver-signup`의 기존 row 연결 실패를 명시적으로 처리하고 롤백 경로를 강화
5. `rate-limit.ts`를 분산 rate limiter로 교체
6. 인코딩이 깨진 주석/문자열을 우선 핵심 서비스 파일부터 정리

## 12. 최종 판단

- 현재 제품은 기능 폭이 넓고, 내부 시연과 제한된 테스트는 가능한 수준이다.
- 다만 `가입/결제 데이터 일관성`, `signed URL 저장`, `무결성 검사 정합성`, `기사 가입 원자성`은 파일럿 전에 손보는 편이 안전하다.
- 이번 정리로 AI 프롬프트와 검토 문서 기준점은 명확해졌고, 이제 남은 일은 운영성 높은 핫스팟을 집중 보완하는 것이다.

## 🔄 미해결

### 1. Amendment 알림 UX
- **무엇:** 기사 입장에서 계약 조건이 중간에 바뀌었을 때 보이는 모바일 화면 디자인.
- **왜:** 푸시 알림 본문, amendment 상세 화면, approve/reject 흐름의 UX가 정의되지 않으면 정산일에야 변경을 알게 되어 기사 신뢰가 깨진다.
- **현황:** schema와 service는 존재(`contract_amendments` 테이블, `amendment.service.ts`). 모바일 라우트 `mobile/app/amendment/[id].tsx`는 있지만 디자인 미확정.
- **선행:** Contract amendments UI (이미 portal+mobile 양쪽 존재).

### 2. PortOne KISA 본인인증 운영 리스크
- **무엇:** KISA 컴플라이언스 상태, 통신사 약정 승인, PortOne 계좌 한도 확인.
- **왜:** PortOne V2 본인인증은 사업자등록 + KISA 컴플라이언스 + 통신사 약정 승인이 필요. 코드 변경이 아니라 승인 프로세스 — PortOne이 거절하면 서명 자체가 막힌다.
- **현황:** 코드는 통합됨. 운영팀이 승인 트랙 관리 필요.
- **블록:** 외부 — 개발과 병렬 진행 가능.

### 3. 오프라인 인디케이터 디자인
- **무엇:** 기사 폰이 신호 없을 때 보이는 화면.
- **왜:** MMKV 캐싱으로 계약/정산 조회는 가능하지만 서명은 네트워크 필수. 인디케이터 없으면 신호 약한 곳에서 서명 시도하다 모호한 에러를 만난다.
- **현황:** 상단 배너 "오프라인 상태입니다", 비활성화된 서명/검증 버튼 + 툴팁, 캐시된 콘텐츠 조회 가능.
- **선행:** 모바일 앱 코어 화면.

### 4. 결제 ↔ 구독 단일 트랜잭션 통합
- **무엇:** 가입 직후 카드 결제 성공/실패와 `subscriptions` row 생성을 단일 트랜잭션으로 묶기.
- **왜:** 현재는 `/api/auth/signup`이 free 플랜 구독 row를 만들고, 이후 별도 엔드포인트(`/api/payment`)에서 결제 수단 등록 → 구독 갱신. 결제 실패 시 `pending_payment` 상태로 남는 케이스 등 분리 흐름이라 운영팀이 수동 조회/조정해야 함.
- **현황:** 보고서 §5-4의 "브라우저에서 직접 insert" 부분은 이미 서버화됨. 다음 단계는 결제 콜백과 구독 상태를 Postgres 트랜잭션 또는 webhook 큐로 묶기.
- **선행:** 첫 카드 결제 고객사 1건 흐름 관찰 후 진행.

### 5. 인코딩 드리프트 정리
- **무엇:** 일부 한국어 주석/문자열이 깨진 채 남아 있는 파일들.
- **왜:** 운영 중 치명 버그는 아니나 QA/유지보수 속도를 늦춘다.
- **예시 파일:** `web/src/services/signed-pdf.service.ts`, `web/src/services/integrity-check.service.ts`, 일부 테스트.
- **블록:** 없음. 시간 날 때마다 단발성 PR로 처리 가능.

### 6. E2E 시나리오 자동화
- **무엇:** 가입 → 결제 → 구독 반영 → 계약 발송 → 모바일 서명 → signed PDF → 무결성 검사 전체 흐름의 자동 회귀 테스트.
- **왜:** 단위 테스트는 28개 통과 중이지만 통합 E2E는 없음. 보고서 §8 지적 사항.
- **선행:** 안정화 단계 완료 후 운영 안정화 패키지 단계로 분류.
